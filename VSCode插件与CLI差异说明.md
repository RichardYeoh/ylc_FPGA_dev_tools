# VSCode 插件与 CLI 使用差异说明

## 1. 当前结论

当前在 VSCode 插件中的 Codex 会话，已经继承了 CLI 侧共用的全局配置目录 `C:\Users\liche\.codex` 中的核心设置。

已确认生效的配置来源包括：

- `C:\Users\liche\.codex\AGENTS.md`
- `C:\Users\liche\.codex\config.toml`
- `C:\Users\liche\.codex\memories\MEMORY.md`
- `C:\Users\liche\.codex\memories\GLOBAL_SESSION_INDEX.md`

其中已经确认继承到的用户偏好包括：

- 默认使用中文回答
- 偏向 FPGA / Verilog / SpinalHDL 语境
- 项目目录内普通非破坏性文件操作可直接进行
- Vivado 优先使用较新的已安装版本
- 代码注释要求中英双语

## 2. 共同点

VSCode 插件与 CLI 在以下方面是共通的：

- 都会读取全局配置目录下的 `AGENTS.md`、`config.toml` 和 `memories`
- 都会叠加当前项目中的 `AGENTS.md`
- 都会受到当前会话中的用户明确指令影响
- 都可以复用全局记忆、项目规则和既有工作习惯

可简单理解为：

`全局规则 + 项目规则 + 当前任务要求 = 当前会话行为`

## 3. 主要差异

### 3.1 VSCode 插件

特点：

- 更贴近当前打开的工作区（workspace）
- 更适合边看代码边修改
- 更容易稳定命中当前项目的本地规则
- 对多文件联动查看更方便

注意点：

- 需要先确认当前打开的 workspace 根目录是否正确
- 如果 VSCode 同时打开多个目录，要明确当前任务针对哪个根目录
- 命令执行结果通常由助手转述，不一定像 CLI 那样直接暴露完整终端链路
- 切换工作区后，项目级规则的命中对象也会随之变化

### 3.2 CLI

特点：

- 更贴近原始终端环境
- 更适合批处理、脚本执行、日志排查和构建任务
- 当前目录、环境变量、shell 行为对结果影响更直接

注意点：

- 从哪个目录启动，往往直接影响项目规则是否命中
- 如果 `cwd` 不对，搜索、修改、构建都可能落到错误位置
- 更适合强命令链路任务，但也更容易因为路径不准导致误操作

## 4. 实际使用建议

建议在 VSCode 插件中遵循以下习惯：

- 开始任务前先确认当前 workspace 根目录
- 涉及子项目时，显式说明目标目录
- 需要记录会话时，直接明确提出
- 涉及构建、脚本、批量处理时，必要时说明希望按 CLI 风格处理

建议在 CLI 中遵循以下习惯：

- 开始任务前先确认当前 `cwd`
- 明确是针对项目根目录还是某个子目录执行
- 对重要命令尽量附带目标路径上下文
- 在跨项目切换时，重新确认当前命中的项目规则

## 5. 推荐的沟通方式

为了减少路径和上下文误判，推荐在任务开始时直接说明以下信息：

- 当前目标目录
- 是否允许直接修改文件
- 是否需要记录会话
- 是否要执行构建、脚本或工具链命令

例如：

> 以当前 workspace 根目录为准，直接修改文件，不需要记录会话。

或：

> 目标是 `text_code_to_utf8` 子目录，本次需要同步记录会话，并执行本地验证。

## 6. 一句话总结

VSCode 插件与 CLI 不是两套完全独立的配置体系，而是优先共享同一套全局规则；真正最需要注意的差别，主要在于“当前工作目录 / 工作区边界”以及“命令执行链路的可见性”。

## 7. FPGA / Vivado 项目特别注意事项

对于 FPGA、Vivado、仿真脚本、约束文件和工程目录，VSCode 插件与 CLI 的使用差异会更敏感，建议额外注意以下几点。

### 7.1 工程根目录与子工程边界

在 FPGA 项目中，经常同时存在：

- RTL 源码目录
- 仿真目录
- Vivado 工程目录
- 生成脚本目录
- 约束文件目录
- IP 输出目录

因此无论在 VSCode 插件还是 CLI 中，都应尽量明确当前操作对象到底是：

- 项目根目录
- 某个子工程目录
- 某个 `runs` / `sim` / `scripts` 目录

否则容易出现以下问题：

- 改到了错误的约束文件
- 在错误目录下启动 Vivado
- 把脚本输出生成到非预期位置
- 把临时文件和工程文件混在一起

### 7.2 VSCode 更适合的 FPGA 类任务

VSCode 插件更适合以下任务：

- 阅读和联动修改 Verilog / SpinalHDL / Tcl / Python 脚本
- 对比多个模块接口与时序逻辑
- 修改约束文件、自动化脚本和文档
- 在同一工作区中同时查看源码、日志和说明文件

但要注意：

- VSCode 中看到的是“编辑视角”，不等于当前 shell 的实际执行目录
- 如果 workspace 中同时挂了多个工程，要显式点名目标工程
- 对生成类目录不要仅凭文件树视觉判断，应结合路径确认

### 7.3 CLI 更适合的 FPGA 类任务

CLI 更适合以下任务：

- 调用 Vivado / xsim / xvlog / xelab / 第三方仿真器
- 跑批处理 Tcl
- 做日志筛查和批量文本检索
- 执行构建、综合、实现、导出、比对等流程型任务

但要注意：

- 一旦当前目录不对，工具可能引用错误的工程文件或相对路径资源
- Tcl、bat、ps1、make 等脚本通常对相对路径非常敏感
- 某些生成目录体积很大，误跑一次实现可能带来大量中间文件

### 7.4 关于生成文件和可清理目录

FPGA 工程中常混有大量“源码”与“生成物”，这类任务在两种方式下都应特别小心区分：

- 应谨慎修改：RTL、约束、Tcl、脚本、文档
- 应谨慎删除：工程缓存、仿真输出、实现输出、IP 生成物

尤其在 VSCode 插件里，文件树操作很直观，但并不代表该目录就是可安全清理的。

对于 Vivado 工程，通常应先区分：

- 可重建的生成目录
- 需要保留的源文件目录
- 由上游脚本自动生成、但仍需纳入版本管理的半生成文件

### 7.5 推荐工作方式

对于 FPGA / Vivado 项目，推荐采用以下分工方式：

- 在 VSCode 插件里完成：代码阅读、局部修改、文档更新、结构梳理
- 在 CLI 里完成：构建、仿真、综合、实现、日志提取、批处理验证

如果希望由助手协助，最好在任务开始时直接说明：

- 目标工程路径
- 是只改源码，还是需要跑工具链
- 是否允许触碰生成目录
- 是否需要保留或记录构建日志

例如：

> 以 `DIPM_IPB_sar_plin6` 为目标工程，只改 RTL 和 Tcl，不清理生成目录，不跑实现。

或：

> 目标是当前工程根目录，需要检查约束和时序相关 Tcl，并在必要时调用 Vivado 做验证。

## 8. 推荐指令模板

下面这些表达方式适合直接在 VSCode 插件或 CLI 中复用，有助于减少路径、权限和任务边界上的歧义。

### 8.1 只读分析

适用于先看代码、先定位问题、不立即改文件的情况。

模板：

> 以当前 workspace 根目录为准，先只读分析，不修改文件。帮我定位这个问题的可能原因，并指出相关文件。

FPGA 示例：

> 以当前工程根目录为准，先只读分析，不改文件。帮我检查这个时序告警可能与哪些 RTL、XDC 和 Tcl 有关。

### 8.2 允许直接修改源码

适用于已经确定要改代码或脚本的情况。

模板：

> 以当前 workspace 根目录为准，允许直接修改项目内文件。先定位问题，再直接修复，并说明改动点。

FPGA 示例：

> 目标是当前子工程，允许直接修改 RTL 和 Tcl。先定位复位时序问题，再直接修复并说明影响范围。

### 8.3 限定修改范围

适用于不希望助手碰到生成目录或其他高风险区域的情况。

模板：

> 只允许修改源码和文档，不要改生成目录、缓存目录和第三方依赖。

FPGA 示例：

> 只允许修改 `src`、`rtl`、`sim`、`constraints` 和脚本文件，不要改 `runs`、`cache`、`ip_user_files` 一类生成目录。

### 8.4 指定目标子目录

适用于一个 workspace 里有多个子项目的情况。

模板：

> 本次只处理 `text_code_to_utf8` 子目录，其他目录只可参考，不要修改。

FPGA 示例：

> 本次只处理 `DIPM_IPB_sar_plin6`，不要改 `plin1` 或其他平行工程。

### 8.5 需要执行验证但不跑重任务

适用于想做轻量验证、不希望直接跑综合实现的情况。

模板：

> 修改完成后做轻量验证即可，不要执行耗时很长的完整构建。

FPGA 示例：

> 修改完成后只做语法检查、脚本检查和必要的日志核对，不跑 synth / impl。

### 8.6 允许调用工具链

适用于确实希望我执行 Vivado、仿真器或相关脚本的情况。

模板：

> 如有必要，可以直接调用本地工具链做验证，并把关键结果总结给我。

FPGA 示例：

> 如有必要，可以调用 Vivado 执行 Tcl 检查和工程验证，但先不要清理已有生成目录。

### 8.7 明确需要记录会话

适用于你希望沉淀项目过程和后续可追溯的情况。

模板：

> 本次任务请记录会话，并同步更新相关项目记录。

FPGA 示例：

> 这次关于时序排障的分析和修改请记录会话，后续我还要继续复盘。

### 8.8 适合当前工作流的一条完整模板

如果你想一次把边界说清，可以直接用这一类完整表达：

> 以当前 workspace 根目录为准，目标子工程是 `xxx`。允许直接修改源码、脚本和文档，不要动生成目录。先分析再修复，修复后做轻量验证，不跑完整实现。本次任务需要记录会话。
