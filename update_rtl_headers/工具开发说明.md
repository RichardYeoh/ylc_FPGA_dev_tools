# 工具开发说明

## 1. 工具概述

本工具由两个入口组成：

- 核心脚本 [update_rtl_headers.py](/E:/proj/ag_other/sources_1/update_rtl_headers.py)
- 一键执行入口 [run_update_rtl_headers.bat](/E:/proj/ag_other/sources_1/run_update_rtl_headers.bat)

其中：

- Python 脚本负责实际处理逻辑
- BAT 文件负责面向最终用户的“一次双击直接执行”体验

当前支持处理的文件扩展名：

- `.v`
- `.vh`
- `.sv`
- `.svh`

工具目标：

- 删除文件头中的 `Copyright` / `CopyRight` 行
- 删除紧随版权声明后的保密声明行，如 `Confidential Proprietary`、`All Rights Reserved`
- 删除文件头中的 `AUTHOR'S EMAIL` / `Email` 行
- 按规则替换 `AUTHOR` / `Author` / `Author/Designer` 字段内容
- 按规则替换 `Release history` 表格中 `AUTHOR` 列内容
- 保持其他注释内容、正文逻辑和非目标作者信息不变

## 2. 设计原则

### 2.1 只处理文件头

脚本通过 `split_header()` 将每个文件拆分为：

- 文件头连续注释区
- 文件正文

只对文件开头连续出现的空行和注释行进行处理，一旦遇到第一个非注释、非空行，就停止头部扫描。这样可以避免误改模块正文中的普通注释。

### 2.2 尽量保留原排版

对于普通 `AUTHOR` 字段，脚本尽量保留原注释前缀和标签风格，例如：

- `// AUTHOR : xxx`
- `// Author       : xxx`
- `//Author/Designer : xxx`
- `//      Author             : xxx`

对于 `Release history` 表格，脚本通过列坐标替换作者列内容，而不是简单按空格切分，这样可以尽量保留上下对齐效果。

### 2.3 仅替换符合规则的作者

当前版本不是把所有作者都强制改为 `ylcheng`，而是只在作者内容与目标模式高度相似时才替换。

## 3. 当前规则

### 3.1 删除规则

以下内容会被删除：

- 文件头中的 `Copyright` / `CopyRight` 行
- 紧跟版权行之后、命中以下关键词的保留声明行：
  - `confidential`
  - `proprietary`
  - `all rights reserved`
- 文件头中的邮箱字段行：
  - `AUTHOR'S EMAIL`
  - `Email`

### 3.2 作者字段替换规则

适用范围：

- 普通字段：
  - `AUTHOR`
  - `Author`
  - `Author/Designer`
- `Release history` 表格中的 `AUTHOR` 列

替换目标值固定为：

```text
ylcheng
```

替换条件：

1. 先将原作者内容转成小写
2. 去掉所有非字母字符
3. 分别与以下目标串比较：
   - `yanglicheng`
   - `lichengyang`
4. 采用“字母多重集交集计数”方法，不考虑字母顺序
5. 若与任一目标串的公共字母数大于 8，则判定为应替换

例如以下内容会被替换：

- `Yang Licheng`
- `YangLicheng`
- `lichengyang`
- `ylcheng@actri.avic`

以下内容不会被替换：

- 空作者字段
- 与目标名字相似度不足的其他英文名
- 中文姓名
- 其他不相关字符串

## 4. 核心实现说明

### 4.1 文件遍历

`iter_source_files()` 递归扫描目标目录下所有受支持扩展名文件，并按路径排序，保证执行稳定。

### 4.2 文件头切分

`split_header()` 负责提取文件开头连续注释区。识别为头部内容的行包括：

- 空行
- `//` 注释
- `/*`
- `*`
- `*/`

### 4.3 单行注释处理

`maybe_rewrite_comment_line()` 处理三类内容：

- 删除邮箱字段
- 规范普通作者字段
- 删除版权字段

普通作者字段通过正则识别标签和原值区域，若 `should_replace_author()` 返回 `True`，则改写为：

```text
: ylcheng
```

### 4.4 Release history 表格处理

`transform_header()` 在扫描到如下表头时启用表格模式：

```text
// VERSION    Date          AUTHOR            DESCRIPTION
```

处理逻辑：

1. 找出 `AUTHOR` 列起始位置
2. 找出 `DESCRIPTION` 列起始位置
3. 对后续符合版本行格式的记录行，截取这两个列坐标之间的作者字段
4. 若命中替换规则，则只替换该列内容为 `ylcheng`
5. 用左对齐方式填回原列宽

版本行识别格式为：

```text
// <version> <yyyy-mm-dd> <author> <description>
```

### 4.5 作者匹配算法

`should_replace_author()` 的实现逻辑：

1. 使用 `re.sub(r"[^a-z]", "", value.lower())` 归一化字符串
2. 使用 `collections.Counter` 统计字母频次
3. 与 `yanglicheng` 和 `lichengyang` 的字母频次做交集
4. 交集字母数大于 8 即命中

该方法优点：

- 不依赖单词顺序
- 可容忍空格、下划线、连字符、邮箱分隔符等噪声

该方法风险：

- 规则较宽，某些非目标但高度相似的英文串也可能被替换

## 5. 执行入口设计

### 5.1 Python 脚本

Python 脚本用于：

- 命令行检查
- 命令行写回
- 供 BAT 入口调用

当前输出已补充为：

- 当前模式
- 处理根目录
- 扫描到的源文件总数
- 命中的文件数
- 命中文件路径列表

示例输出：

```text
[APPLY] root: E:\proj\ag_other\sources_1\rtl
[APPLY] scanned source files: 36
[APPLY] updated 26 file(s)
```

### 5.2 BAT 一键入口

`run_update_rtl_headers.bat` 的设计目标是让普通用户无需打开终端、无需手工输入命令、无需分步确认，直接双击即可完成一次处理。

执行流程固定为：

1. 切换到脚本所在目录
2. 执行实际改写
3. 自动执行一次复测
4. 在窗口中保留完整输出
5. 使用 `pause` 等待用户关闭窗口

这样设计的原因：

- 降低使用门槛
- 避免用户只做检查而忘记真正写回
- 让用户在关闭窗口前看到完整处理结果

### 5.3 无确认模式

当前一键入口不会弹出“是否继续”确认。

这是本轮需求明确要求的行为，因此：

- 双击后会直接改写 `rtl/` 中命中的文件
- 说明文档中必须明确提醒用户此行为是直接写回
- 若需要保守操作，应改用 Python 脚本检查模式

## 6. 命令行接口

脚本命令格式：

```powershell
python .\update_rtl_headers.py [root]
python .\update_rtl_headers.py --apply [root]
```

参数说明：

- `root`
  - 可选
  - 默认值为 `rtl`
  - 表示待处理目录
- `--apply`
  - 省略时仅检查，不写回
  - 带上时执行实际写回

输出格式：

- 检查模式：

```text
would update N file(s)
```

- 写回模式：

```text
updated N file(s)
```

后续会逐行输出命中的文件路径。

## 8. 已知边界

- 只处理文件开头连续注释区，不处理正文中的零散注释
- `Release history` 表格必须包含标准表头 `VERSION Date AUTHOR DESCRIPTION` 才会进入列模式
- 作者匹配规则是“高相似度”而非“严格等于”，因此存在误命中的可能
- 当前仅支持单文件单表头格式，不支持复杂嵌套注释结构
- 文件读取使用 `utf-8` 且 `errors="ignore"`，若源文件编码异常，可能跳过不可解码字符
- 一键 BAT 入口会直接写回，不包含二次确认
- Windows 控制台若编码设置异常，终端中的中文预览可能乱码，但不影响文件本身写入
