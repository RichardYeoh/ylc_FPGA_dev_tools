# OpenClaw对标记忆重构方案

## 结论

- 目前仍建议使用显式触发词来开启和维护记录。
- 原因不是 Markdown 不能写，而是宿主层仍没有稳定的每轮自动落盘钩子。
- 当前最稳的方案仍是：
  - 项目本地全文记录作为真源
  - 全局记忆按分层结构维护
  - 通过少量高频触发词显式控制记录与 flush

## 最常用触发词

- `开启记录模式`
  用途：开启当前项目的记录。默认写项目本地全文记录。

- `开启记录模式，同步全局`
  用途：开启当前项目记录，并允许本次工作同步进入全局记忆层。

- `结束前补全记录`
  用途：会话收尾时统一补写项目记录，并补齐需要的全局记忆。

## 推荐触发词

- `把这段追加到会话记录`
  用途：只补写当前项目本地全文记录。

- `flush全局记忆`
  用途：把当前阶段已经稳定的信息写入全局记忆层。

## 兼容维护命令

- `执行全局记录维护`
- `整理全局记录`
- `压缩全局记录`
- `更新全局索引`
- `整理经验卡片`

说明：
- 这些命令继续保留，但属于维护动作，不建议作为日常高频入口。
- 日常使用优先前三条最常用触发词。

## 简化后的使用建议

### 1. 会话开始

- 常规项目：`开启记录模式`
- 本次工作明确希望进入全局层：`开启记录模式，同步全局`

### 2. 工作进行中

- 需要补记当前一段过程：`把这段追加到会话记录`
- 形成了稳定决策、阶段结果或可复用经验：`flush全局记忆`

### 3. 会话结束

- 使用：`结束前补全记录`

## 为什么仍不建议默认每轮自动记录

- 宿主没有稳定的 post-turn hook。
- 自动逐轮写盘容易写错文件、写重复内容、写入低价值临时对话。
- 每轮都做语义判断和整理会增加额外 token 消耗。
- 当前“显式触发 + 阶段性补写 + 收尾补全”的方案，稳定性明显更高。

1. `GLOBAL_SESSION_INDEX.md`
2. `MEMORY.md`
3. `GLOBAL_EXPERIENCE_CARDS.md`
4. 相关日期的 `memory\YYYY-MM-DD.md`
5. 项目本地全文会话记录

## 写入分流

### 写入项目本地全文记录

- 原始对话
- 任务过程
- 用户与助手的详细来回

### 写入 `MEMORY.md`

- 长期稳定偏好
- 长期工程约束
- 长期有效工作规则

### 写入 `memory\YYYY-MM-DD.md`

- 当天的重要决策
- 阶段性结果
- 尚未晋升为长期记忆的有效信息

### 写入 `GLOBAL_EXPERIENCE_CARDS.md`

- 可复用工作流
- 可复用排障路径
- 可复用模板与方案
