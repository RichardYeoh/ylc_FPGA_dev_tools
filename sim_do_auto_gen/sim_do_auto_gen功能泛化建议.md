# sim_do_auto_gen功能泛化建议

## 1. 结论先行

对照根目录说明文档、`20260407` 实战会话记录以及新旧 `gen_my_sim.ps1` 差异后，可以确认当前工具的真正能力边界已经超过“上一级生成 `my_sim.tcl/my_sim.bat`”这一单点功能。

更准确的定位应该是：

`sim_do_auto_gen` 是一个面向 `Vivado` 导出仿真脚本的二次包装生成器，用来在不直接修改原始导出脚本的前提下，补齐运行目录、用户保留区、启动方式以及运行时外部资源等问题。

如果要继续提升通用性，建议把泛化方向聚焦到“包装层能力抽象”，而不是继续围绕 `questa/modelsim` 目录做更多特判。

## 2. 本次对照依据

本次分析基于以下材料：

- 根目录文档：
  - `仿真do脚本自动生成.md`
  - `工具开发说明书.md`
  - `用户指导手册.md`
  - `团队培训精简手册.md`
- 实战记录：
  - `20260407/sim_do_auto_gen/v_hdmi_tx_ss_0_ex_818sim会话记录.md`
- 新旧脚本差异：
  - 根目录 `gen_my_sim.ps1`
  - `20260407/sim_do_auto_gen/gen_my_sim.ps1`

## 3. 当前版本的固定假设

从文档和代码同时看，当前版本绑定了以下工程假设：

| 维度 | 当前假设 | 影响 |
| --- | --- | --- |
| 仿真器 | 仅面向 `Questa/ModelSim` | 工具命名、提示文案、目录选择都被绑定 |
| 输入目录 | 用户手工选择 `behav/questa` 或 `behav/modelsim` | 不利于自动发现、多仿真集和批处理 |
| 目录拓扑 | 输出固定放上一级 | 适合当前工程，但不一定适合所有团队目录规范 |
| 脚本来源 | 强依赖 `*_compile.do`、`*_elaborate.do`、`*_simulate.do`、`simulate.bat` 成套存在 | 对导出脚本变体容错有限 |
| 保留区 | 只保留波形块和最后一条 `run ...` | 用户自定义启动逻辑承接能力有限 |
| 资源补齐 | 根目录版本默认没有抽象成通用能力 | 实战中遇到外部文件依赖时需要临时补洞 |
| 交互方式 | UI 优先，CLI 仅基础参数 | 团队批量复用和 CI/回归接入能力偏弱 |

## 4. 实战记录暴露出的真正泛化点

`20260407` 的实战记录非常关键。它说明当前工具在真实工程里遇到的问题，并不是简单的“某个工程少拷了几个 `.bin` 文件”，而是：

1. `Vivado` 导出的仿真脚本只保证基本的编译、展开、启动链路。
2. 普通 RTL 里的运行时外部资源，例如 `$readmemb/$readmemh` 依赖的初始化文件，并不会被自动兜底。
3. 用户真正需要的是“生成器自动识别这些外部依赖，并在启动前完成 staging”。

这意味着应该把新增能力从“`readmem` 特判”上升为：

`运行前资源扫描 + staging 框架`

这样才能继续扩展到：

- `.bin/.mem/.hex/.mif` 等初始化文件
- 通过相对路径引用的查找表文件
- 后续可能出现的额外 `do/udo` 依赖
- 团队约定的仿真前预处理文件

## 5. 建议的功能泛化方向

### 5.1 P0: 把“目录识别”泛化为“仿真上下文识别”

当前 `Get-SimulatorContext` 的实质判断条件，其实不是“目录名必须叫 `questa/modelsim`”，而是“该目录里是否存在一套可用的导出脚本”。

建议改为以下思路：

- 核心输入从“`QuestaDir`/`SimulatorDir`”升级为“`SimulationScriptDir`”
- 目录合法性以文件存在性判断为主：
  - `*_compile.do`
  - `*_elaborate.do`
  - `*_simulate.do`
  - `simulate.bat`
- `questa/modelsim` 只保留为默认提示，不作为核心逻辑绑定
- 增加自动发现模式：
  - 从用户指定的上层目录自动搜索匹配目录
  - 支持列出多个候选项供用户选择

这样做的收益是：

- 目录名变化时不需要重写工具
- 后续支持更多 `Vivado` 导出层级时，主流程不用大改

### 5.2 P0: 把“`readmem` 补丁”泛化为“运行资源扫描与 staging”

这是本轮最值得做成通用能力的一项。

建议把当前 `20260407` 中新增的 `$readmemb/$readmemh` 处理进一步抽象为：

- `Get-CompileSourceFilesFromLines`
  - 负责从编译脚本里解析真实源码文件
- `Get-RuntimeResourceItems`
  - 负责扫描源码中的运行期外部资源引用
- `Emit-StagingSection`
  - 负责把待同步资源写入生成的 `my_sim.tcl`

首版扫描规则可以先覆盖：

- `$readmemb("...")`
- `$readmemh("...")`

然后保留扩展接口，后续可加入：

- 额外资源文件扩展名白名单
- 用户自定义 staging 列表
- 通过配置文件追加资源映射

这样做后，工具的定位会从：

“TCL/BAT 生成器”

升级为：

“仿真包装脚本生成器 + 运行依赖补齐器”

### 5.3 P0: 输入接口从“隐式推断”扩展到“显式指定”

当前脚本默认取最新的 `*_compile.do` 再推导同名 `*_elaborate.do` 和 `*_simulate.do`。这对单一仿真集足够，但在多套导出结果并存时容易误选。

建议增加可选显式参数：

- `-CompileDoPath`
- `-ElaborateDoPath`
- `-SimulateDoPath`
- `-SimulateBatPath`
- `-OutputDir`

默认仍保留自动推断，但允许高级用户精确指定。

这样做后：

- 多 `sim set` 工程更好用
- 自动脚本调用更可控
- 出问题时更容易排障

### 5.4 P1: 把“输出固定上一级”泛化为“输出布局策略”

当前“输出到上一级再切回仿真目录执行”的设计是合理的默认值，但不应该是唯一布局。

建议增加布局模式参数，例如：

- `parent`
  - 默认模式，保持当前行为
- `inplace`
  - 直接在仿真脚本目录生成包装脚本
- `custom`
  - 输出到用户指定目录，并在脚本中记录真实仿真目录

这样能覆盖不同团队的目录管理习惯，也更适合做共享启动入口。

### 5.5 P1: 把“只保留波形块和 run 行”泛化为“用户可保留区”

当前用户保留策略偏窄，只能继承：

- 用户波形块
- 最后一条 `run ...`

建议扩展为多个命名保留区，例如：

- 用户波形区
- 用户预启动命令区
- 用户后启动命令区
- 用户运行控制区
- 用户注释区

实现方式不一定复杂，可以继续沿用标记区模型，只是改成多段命名锚点。

这样做的好处是：

- 用户能加自己的 `do`、`force`、`radix`、`log`、`when` 等操作
- 重生成时不容易丢团队自定义逻辑

### 5.6 P1: 把“手工 UI 工具”泛化为“UI + CLI + 配置文件”三入口

当前 UI 适合单工程使用，但团队想复用时会遇到两个限制：

- 多工程重复点选效率低
- 无法稳定接入回归脚本

建议提供三种并存入口：

- UI 模式
  - 保持当前易用性
- CLI 模式
  - 适合脚本化调用
- 配置文件模式
  - 例如 `sim_do_auto_gen.json`
  - 固化目录、输出布局、保留区策略、附加 staging 规则

这样工具才真正具备“团队共享”的通用性。

### 5.7 P2: 增加生成报告与风险提示

当前工具生成成功后，用户只能看到“已生成”，但不知道工具自动做了哪些判断。

建议增加一个可选报告，例如：

- 识别到的导出脚本路径
- 识别到的运行资源数量
- 找到但无法解析的疑似资源引用
- 采用了哪个旧版 `my_sim.tcl` 作为保留区来源
- 最终输出布局模式

报告既可以显示到日志框，也可以输出为一个 Markdown 或 TXT 摘要。

这对排障和团队培训都很有帮助。

## 6. 建议的实现分层

建议后续把脚本内部逻辑按下面三层重构：

### 6.1 上下文层

负责：

- 识别仿真脚本目录
- 解析导出脚本集合
- 确定输出目录与布局模式

建议输出统一的上下文对象，例如：

- `ScriptRootDir`
- `CompileDo`
- `ElaborateDo`
- `SimulateDo`
- `SimulateBat`
- `OutputDir`
- `SimulatorLeaf`
- `LayoutMode`

### 6.2 分析层

负责：

- 解析源码文件列表
- 扫描外部资源依赖
- 识别用户保留区来源
- 生成 warning 列表

这是最适合继续泛化的层。

### 6.3 生成层

负责：

- 输出 `my_sim.tcl`
- 输出 `my_sim.bat`
- 输出可选报告
- 输出提示文案

这样以后想增加新能力时，不必频繁改 UI 主流程。

## 7. 推荐的落地优先级

建议按下面顺序推进：

### 第一阶段

- 合并 `20260407` 中的资源 staging 能力回主版本
- 将该能力正式写入开发文档和用户文档
- 把目录判断从“文案绑定 `questa/modelsim`”调整为“逻辑以脚本集合存在性为准”

### 第二阶段

- 增加显式输入参数
- 增加输出布局模式
- 增加更多用户保留区

### 第三阶段

- 增加自动发现候选目录
- 增加配置文件模式
- 增加生成报告

## 8. 我认为最值得优先做的三项

如果只先做三项，我建议优先：

1. 运行资源扫描与 staging 泛化  
   这是已经被真实工程验证过的痛点，收益最大。

2. 显式输入参数与更稳健的上下文识别  
   这能直接降低多仿真集工程里的误用风险。

3. 多保留区机制  
   这能显著提升重复生成时的可维护性，减少用户二次手改流失。

## 9. 对现有文档的修订建议

根目录四份文档现在都偏强调：

- `questa/modelsim` 固定目录
- `behav` 上一级输出
- “不要移回去”这一特定使用法

后续建议统一改成下面这种口径：

- 工具主定位：仿真包装脚本生成器
- 默认工作流：针对 `Vivado` 导出的 `Questa/ModelSim` 脚本
- 默认布局：输出到上一级
- 可扩展点：资源 staging、保留区、布局模式、配置文件模式

这样文档才能跟后续演进保持一致，不会再出现“实战能力已经增加，但文档仍停留在旧模型”的割裂。
