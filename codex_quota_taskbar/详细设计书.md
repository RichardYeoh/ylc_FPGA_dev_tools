## 2. 设计目标

本工具用于在 Windows 任务栏区域紧凑显示 Codex/GPT Codex 配额状态，重点显示 `5h` 窗口余量、weekly 周窗口余量，以及各自复位时间。工具需要满足以下目标：

- 显示面积小，默认停靠在 Windows 任务栏系统托盘左侧。
- 不拦截任务栏、托盘、网络、音量、时钟等区域点击。
- 在打开 Codex CLI、VS Code Codex/OpenAI/ChatGPT 扩展宿主或桌面相关进程时自动启动显示窗。
- Codex 相关进程全部退出后，按配置自动关闭显示窗。
- 优先读取真实 Codex usage API；API 失败时保留缓存和手工状态文件回退能力。
- 不依赖第三方 Python GUI 包，便于在普通 Windows Python 环境中直接运行。

## 3. 范围与边界

### 3.1 范围内功能

- Windows 任务栏文字提示窗。
- Codex usage API 数据读取、标准化和缓存。
- 外部 provider command 扩展入口。
- watcher 后台进程检测和联动启动/关闭。
- 当前用户登录后自动启动 watcher。
- 基础 smoke test 验证。

### 3.2 范围外功能

- 不实现 Windows COM DeskBand 或正式任务栏插件。
- 不修改 OpenAI/Codex 官方配额策略。
- 不把本地 token 统计伪装为官方配额。
- 不提供跨平台 UI，当前实现依赖 Windows `user32`、Tkinter 和 PowerShell。

## 5. 总体架构

系统由三个主要运行单元组成：

| 单元 | 文件 | 职责 |
| --- | --- | --- |
| 配额数据 provider | `src/codex_quota_provider.py` | 读取配置、读取 Codex OAuth 状态、访问 usage API、刷新 token、缓存和标准化配额数据 |
| 任务栏显示窗 | `src/codex_quota_taskbar.py` | 创建 Tkinter 紧凑窗口、读取配额快照、格式化显示、计算任务栏位置、保持可见 |
| 进程 watcher | `src/codex_quota_watcher.py` | 周期扫描进程，判断 Codex 是否运行，启动或关闭显示窗 |

PowerShell 脚本负责将这些 Python 入口包装成用户可直接运行的命令，并提供开机自启动安装/卸载能力。

## 6. 数据流设计

### 6.1 正常数据路径

```text
Codex auth.json
  -> refresh/access token
  -> chatgpt.com/backend-api/codex/usage
  -> normalize_api_usage()
  -> quota_cache.json
  -> QuotaSnapshot
  -> Tkinter labels
```

### 6.2 回退数据路径

```text
provider_command
  -> JSON output
  -> QuotaSnapshot
```

如果外部 provider 未配置或失败：

```text
quota_cache.json
  -> QuotaSnapshot
```

如果实时 API 和缓存都不可用：

```text
quota_state.json
  -> QuotaSnapshot
```

## 7. 配额数据模型

内部使用两个 dataclass 表示标准化后的数据：

| 类型 | 关键字段 | 说明 |
| --- | --- | --- |
| `QuotaWindow` | `key`、`label`、`remaining_fraction`、`remaining_text`、`reset_at` | 单个配额窗口 |
| `QuotaSnapshot` | `source`、`updated_at`、`five_hour`、`weekly` | 一次完整配额快照 |

标准 JSON 结构如下：

```json
{
  "updated_at": "2026-06-11T10:00:00+00:00",
  "source": "chatgpt-codex-usage-api",
  "windows": {
    "5h": {
      "remaining_fraction": 0.94,
      "remaining_text": "94%",
      "reset_at": "2026-06-11T12:30:00+00:00"
    },
    "weekly": {
      "remaining_fraction": 0.83,
      "remaining_text": "83%",
      "reset_at": "2026-06-16T03:00:00+00:00"
    }
  }
}
```

## 8. 配置设计

配置文件为 `config.json`，按功能分为 `window`、`quota`、`watcher` 三组。

### 8.1 window

| 字段 | 当前默认值 | 说明 |
| --- | --- | --- |
| `width` | `146` | 固定宽度回退值；启用 `auto_size` 时仅作为备用 |
| `height` | `44` | 固定高度回退值 |
| `auto_size` | `true` | 按 Tk 文本请求尺寸自动收紧窗口 |

| `dock_gap` | `8` | 与托盘区域的基础间距 |
| `dock_offset_x` | `4` | 基于托盘左侧定位后的水平微调 |
| `transparent_background` | `true` | 使用透明键色，仅显示文字 |
| `transparent_color` | `#dceff5` | 透明键色，接近浅色任务栏背景以减轻抗锯齿边缘 |
| `click_through` | `true` | 鼠标穿透，不拦截任务栏点击 |
| `font_family` | `Consolas` | 显示字体 |
| `font_size` | `10` | 显示字号 |
| `text_color` | `#4f5f66` | 正常文字色 |
| `warning_color` | `#7a6048` | 低于 25% 时的告警色 |
| `critical_color` | `#7a4f55` | 低于 10% 时的严重告警色 |

### 8.2 quota

| 字段 | 当前默认值 | 说明 |
| --- | --- | --- |
| `state_file` | `quota_state.json` | 手工状态回退文件 |
| `cache_file` | `quota_cache.json` | API 标准化结果缓存文件 |
| `refresh_seconds` | `2` | 显示窗本地刷新周期 |
| `api_refresh_seconds` | `150` | 实时 API 结果最大缓存时间，也就是查询更新配额的最短间隔 |
| `use_live_api` | `true` | 是否启用 Codex usage API |
| `provider_command` | `[]` | 外部 provider 命令，优先级高于 live API |

### 8.3 watcher

| 字段 | 当前默认值 | 说明 |
| --- | --- | --- |
| `poll_seconds` | `15` | watcher 进程扫描周期 |
| `close_when_codex_exits` | `true` | Codex 进程全部退出后关闭显示窗 |
| `process_patterns` | `codex.exe` 等 | 常规 Codex 进程匹配项 |
| `vscode_patterns` | `code.exe` | VS Code 进程名称匹配项 |
| `desktop_patterns` | `codex.exe`、`chatgpt.exe`、`openai` | 桌面应用匹配项 |

## 9. Provider 详细设计

`codex_quota_provider.py` 的核心职责是把不同来源的数据统一转换为 `QuotaSnapshot`。

### 9.1 配置读取

`load_config(project_root)` 从项目根目录读取 `config.json`。所有路径均相对项目目录，避免脚本从不同工作目录启动时读取错配置。

### 9.2 外部命令 provider

`run_provider_command(project_root, command)` 在项目目录下执行外部命令。命令成功并输出合法 JSON 时，返回该 JSON。该路径优先于 live API，便于后续接入新的官方 CLI 输出或自定义采集脚本。

### 9.3 Codex OAuth 状态读取

`find_codex_auth_file()` 优先使用 `CODEX_HOME`，否则读取当前用户目录下的 `.codex/auth.json`。provider 从其中读取 `access_token`、`refresh_token` 和 `account_id`。

### 9.4 Live API 请求

`fetch_live_usage()` 调用 `https://chatgpt.com/backend-api/codex/usage`。当服务端返回 `401` 或 `403` 时，首次失败会尝试用 refresh token 请求 `https://auth.openai.com/oauth/token` 刷新 access token，然后重试 usage API。

请求超时固定为 `REQUEST_TIMEOUT_SECONDS = 6`，避免 UI 长时间卡在网络请求上。

### 9.5 API 数据标准化

`normalize_api_usage()` 从 API 返回的 `rate_limit.primary_window` 和 `rate_limit.secondary_window` 提取配额窗口：

- `primary_window` 映射为 `5h`。
- `secondary_window` 映射为 `weekly`。
- `used_percent` 转换为 `remaining_fraction = (100 - used_percent) / 100`。
- `reset_at` 按 UTC ISO 字符串写入缓存。

### 9.6 缓存策略

`load_cached_or_live()` 先读取 `quota_cache.json`。如果 `updated_at` 距当前时间未超过 `api_refresh_seconds`，直接使用缓存。当前配置为 `150` 秒。

缓存过期时最多尝试 3 次 live API。全部失败后，如果旧缓存仍可读取，则继续使用旧缓存；否则返回 `None`，由 `load_snapshot()` 回退到 `quota_state.json`。

## 10. UI 详细设计

### 10.1 窗口创建

`QuotaTaskbarWindow` 使用 Tkinter 创建无边框窗口：

- `overrideredirect(True)` 去掉系统边框。
- `-topmost` 保持在前。
- `-alpha` 由配置控制。
- `-transparentcolor` 结合 `transparent_color` 实现透明背景。

窗口由两行三列 Label 组成：

| 行 | 标签 | 余量 | 复位时间 |
| --- | --- | --- | --- |
| 第 1 行 | `5H` | 5 小时窗口余量，界面显示百分比数值但不显示 `%` 符号 | 5 小时窗口复位时间 |
| 第 2 行 | `W` | weekly 余量 | weekly 复位时间 |

### 10.2 时间显示

API 缓存中的复位时间按 UTC 保存。显示层通过 `to_hong_kong_time()` 统一转换到香港时间 `UTC+08:00`：

- 如果复位日期是当天，显示 `HH:MM`。
- 如果不是当天，显示 `MM/DD HH`。
- 空值显示 `--:--`。

### 10.3 颜色策略

`quota_value_color()` 使用两个窗口中最低的剩余额度决定整窗文字颜色：

| 条件 | 颜色来源 |
| --- | --- |
| 无有效余量 | `unknown_color` |
| 最低余量 <= 10% | `critical_color` |
| 最低余量 <= 25% | `warning_color` |
| 其他情况 | `text_color` |

### 10.4 任务栏定位

窗口默认模式为 `taskbar_docked`。定位流程如下：

1. 使用 `FindWindowW("Shell_TrayWnd", None)` 获取任务栏窗口。
2. 使用 `EnumChildWindows` 查找 `TrayNotifyWnd`。
3. 若找到托盘区域，则将窗口放在托盘左侧：

```text
x = tray_left - width - dock_gap + dock_offset_x
y = taskbar_top + (taskbar_height - height) / 2
```

4. 使用 `clamp()` 保证窗口不超出任务栏范围。
5. 如果定位失败，则回退到工作区和任务栏边界计算。

### 10.5 可见性恢复

Windows 的“显示桌面”或任务栏交互可能改变顶层窗口层级。每次刷新和定位后，`ensure_visible()` 会调用：

- `ShowWindow(hwnd, SW_SHOWNOACTIVATE)`
- `SetWindowPos(HWND_TOPMOST, ..., SWP_SHOWWINDOW | SWP_NOACTIVATE)`

这样可以恢复显示，同时避免抢焦点。

### 10.6 单实例控制

主窗口通过命名互斥量 `Local\CodexQuotaTaskbarWindow` 防止重复显示窗。watcher 使用 `Local\CodexQuotaTaskbarWatcher` 防止重复监听器。

## 11. Watcher 详细设计

### 11.1 进程查询

`query_processes()` 通过 PowerShell 查询 `Win32_Process`，导出 `Name` 和 `CommandLine` 的 CSV，再由 Python `csv.DictReader` 解析。

### 11.2 Codex 进程识别

`codex_is_running()` 按三类规则识别：

- 进程名或命令行包含 `process_patterns`。
- VS Code 进程名匹配 `vscode_patterns`，命令行含 `openai`、`codex` 或 `chatgpt`。
- 桌面应用匹配 `desktop_patterns`，并同时含 `codex`、`openai` 或 `chatgpt`。

### 11.3 自身进程排除

因为项目路径本身包含 `codex_quota_taskbar`，watcher 通过 `is_own_tool_process()` 显式排除：

- `codex_quota_taskbar.py`
- `codex_quota_watcher.py`
- `scripts\run_watcher.ps1`
- `scripts\run_app.ps1`

这可以避免工具自身被误判为 Codex 进程。

### 11.4 联动控制

watcher 主循环每 `poll_seconds` 秒执行一次：

```text
query_processes()
  -> codex_is_running()
  -> app_is_running()
  -> codex_seen && !app_seen: launch_app()
  -> close_when_codex_exits && app_seen && !codex_seen: stop_app()
  -> sleep(poll_seconds)
```

## 12. 脚本设计

| 脚本 | 功能 |
| --- | --- |
| `scripts\run_app.ps1` | 停止本项目旧显示窗，然后启动新的显示窗 |
| `scripts\run_watcher.ps1` | 启动 watcher |
| `scripts\stop_app.ps1` | 仅停止本项目显示窗，不停止 watcher |
| `scripts\install_autostart.ps1` | 优先注册当前用户登录计划任务；失败时回退到 Startup 快捷方式 |
| `scripts\uninstall_autostart.ps1` | 删除计划任务和 Startup 快捷方式 |

`run_app.ps1` 和 `stop_app.ps1` 都通过项目根目录限定命令行匹配，避免误停止其他目录中的同名脚本。

## 13. 自启动设计

安装脚本优先使用 Windows 计划任务：

```powershell
Register-ScheduledTask -TaskName CodexQuotaTaskbarWatcher -Trigger (New-ScheduledTaskTrigger -AtLogOn)
```

如果当前权限不足导致计划任务注册失败，则创建当前用户 Startup 文件夹快捷方式：

```text
%APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\CodexQuotaTaskbarWatcher.lnk
```

快捷方式目标为 `pwsh` 或 `powershell`，参数指向 `scripts\run_watcher.ps1`。

## 14. 异常处理策略

| 场景 | 处理方式 |
| --- | --- |
| provider command 失败 | 忽略，继续尝试 live API |
| API token 失效 | 尝试 refresh token 后重试 |
| API 多次失败 | 尝试读取旧 `quota_cache.json` |
| 缓存不可用 | 回退到 `quota_state.json` |
| UI 刷新异常 | 显示 `ERR` 和 `--:--` |
| watcher 进程查询失败 | 返回空进程列表，等待下一轮 |
| 重复启动 | 命名互斥量使后启动实例直接退出 |

## 15. 安全与隐私

- 工具读取本机 Codex OAuth 状态，但不把 token 写入项目缓存或日志。
- `quota_cache.json` 只保存标准化后的余量、复位时间、来源和更新时间。
- `provider_command` 会在项目目录执行，配置时应只使用可信脚本。
- 自启动仅针对当前用户，不要求系统服务级常驻。

## 16. 验证设计

现有基础验证入口为：

```powershell
python -m py_compile src\codex_quota_provider.py src\codex_quota_taskbar.py src\codex_quota_watcher.py src\smoke_test.py
python src\smoke_test.py
```

`smoke_test.py` 覆盖以下关键点：

- provider 可返回 `5h` 和 `weekly` 快照。
- UTC 复位时间能转换为香港时间。
- watcher 能识别本项目 UI 正在运行。
- watcher 不会把本工具自身进程误判为 Codex。

## 17. 已知限制

- 任务栏定位依赖 Windows 内部窗口类名 `Shell_TrayWnd` 和 `TrayNotifyWnd`。
- Tkinter 透明键色在不同主题、缩放比例和抗锯齿条件下可能出现边缘差异。
- usage API 是基于现有 Codex OAuth 状态的非 CLI 公开命令路径，后续官方接口变化时需要调整 provider。
- 计划任务注册可能受权限限制，当前已通过 Startup 快捷方式回退。

## 18. 后续改进建议

- 在 UI 上增加缓存更新时间提示，便于判断数据新鲜度。
- 将 provider 失败原因写入受控 debug log，方便排障。
- 增加配置 schema 校验，避免手工改错 JSON 字段类型。
- 如官方 CLI 后续提供稳定 `usage --json`，优先改为 provider command 或内置 CLI provider。
